# 1.1 余弦检索升级基线

- 检索升级基线提交：`cc0b98c7bff7b413ca186f92161635c6d205099c`
- 稳定 1.0 基线提交：`2d82761631161a796193b5cc90104808456b6799`
- 开发分支：`feature/v1.1-cosine-retrieval`
- 1.0 恢复标签：`v1.0.0-retrieval-backup`
- 当前向量模型：`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`
- 模型向量维度：384（模型卡与配置均声明 384；运行时仍检查每批向量维度一致）
- 当前持久化目录默认值：`./chroma_db`
- 当前 1.0 collection 默认名称：`course_material_chunks`（可被旧变量 `CHROMA_COLLECTION` 覆盖）
- 当前 collection 创建方式：`get_or_create_collection(name=...)`，没有显式指定距离空间，因此 Chroma 使用默认 squared L2
- 当前向量生成：资料片段和查询均使用 `normalize_embeddings=True`
- 当前相似度字段：`similarity_score = 1 / (1 + max(distance, 0))`，它只是旧相关度转换分数，不是真实余弦相似度
- 当前检索入口：普通接口调用 `vector_service.semantic_search`；Agent 引用先调用该函数，再在 `citation_service` 中二次查询 SQL
- 当前 Chroma 版本：`1.5.9`

## 风险基线

1. 归一化向量使 squared L2 与余弦相似度的排序单调等价，但返回的 L2 distance 不能直接解释为余弦距离。
2. 旧 collection 的距离空间无法安全原地修改。
3. 当前 `similarity_score` 名称具有误导性。
4. 普通检索和 Agent 引用的 SQL 校验、结构化结果及阈值逻辑没有统一。
5. 删除向量当前只按 `material_id` 构造 Chroma 条件，缺少显式的用户和课程条件。

## 回滚

代码回滚只需切回原分支或从恢复标签建立分支，不需要删除任何 Chroma 数据：

```powershell
git switch release/1.1.0
# 或创建只读排障分支
git switch -c recovery/v1.0-retrieval v1.0.0-retrieval-backup
```

运行配置回滚时，将 collection 配置切回原 1.0 名称 `course_material_chunks`。不得删除 `course_material_chunks_v1_1_cosine`，也不得清空 `chroma_db`。1.1 期间新上传的资料默认只写入 1.1 collection，回滚后如需检索这些新增资料，应在旧环境中重新建立向量。
