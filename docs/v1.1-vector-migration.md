# 1.0 到 1.1 向量迁移

## 原则

- 旧 collection `course_material_chunks` 是回滚快照，迁移脚本不会修改或删除它。
- 新 collection 为 `course_material_chunks_v1_1_cosine`。
- collection 的距离空间不能安全原地修改，因此必须从 MySQL 片段重新编码。
- 脚本不会修改 `.env`；只有人工校验通过后才能切换。
- 不清空 `chroma_db`，不删除 `MaterialChunk`。

执行前建议备份 MySQL 和整个 `chroma_db` 目录，并确认当前命令使用的是预期数据库。不要在未知或生产数据库上直接执行 `--apply`。

## 1. Dry-run

默认命令只读取 SQL 并统计可迁移片段，不加载模型、不写 Chroma：

```powershell
cd course-agent-backend
.\.venv\Scripts\python.exe scripts\rebuild_cosine_vectors.py
```

或：

```powershell
.\scripts\rebuild_cosine_vectors.ps1
```

资格条件包括：用户、课程和资料存在且归属一致，资料 `parse_status="success"`，片段正文非空。脚本不会信任旧 `vector_status`。

## 2. 重建独立 collection

确认数据库、备份和 dry-run 数量后，显式执行：

```powershell
.\.venv\Scripts\python.exe scripts\rebuild_cosine_vectors.py `
  --apply --yes `
  --collection course_material_chunks_v1_1_cosine `
  --batch-size 64
```

或：

```powershell
.\scripts\rebuild_cosine_vectors.ps1 -Apply -BatchSize 64
```

每条向量 ID 为 `chunk_{chunk_id}`，写入采用 upsert，因此可安全重复执行。metadata 包含 `user_id`、`course_id`、`material_id`、`chunk_id`、`chunk_index`、`page_no`、`file_type` 和 `embedding_version`。

单批失败不会清除前面成功的批次，也不会删除旧 collection。输出包含 eligible、succeeded、failed、skipped、batches 和 failed_chunk_ids。

## 3. 校验

```powershell
.\.venv\Scripts\python.exe scripts\validate_cosine_collection.py `
  --collection course_material_chunks_v1_1_cosine
```

校验内容：

1. collection 配置必须是 cosine。
2. SQL 可向量化片段数与新 collection 记录数一致。
3. 抽样 metadata 字段完整。
4. 每条抽样记录能以用户、课程、资料和片段四级条件匹配 SQL。
5. 使用同一归一化模型执行随机查询。
6. 查询结果不越过用户和课程范围。
7. `similarity_score = 1 - distance` 且位于 `[-1, 1]`。

只有报告 `ok: true` 才建议切换。

## 4. 人工切换

在本地 `.env` 中设置（不要提交该文件）：

```dotenv
CHROMA_COLLECTION_NAME=course_material_chunks_v1_1_cosine
```

若旧 `.env` 仍有 `CHROMA_COLLECTION=course_material_chunks`，新变量优先；确认应用启动和检索正常后可自行移除旧变量。重启 FastAPI/Electron 后生效。

## 5. 增量一致性

切换后，上传、手动重建、重新解析和重新分块只操作当前 1.1 collection：

- 写入成功后才将 SQL `vector_status` 标为 success。
- 删除向量同时限定 `user_id + course_id + material_id`。
- 写入失败会清理当前资料在 1.1 collection 中的部分结果并标记 failed。
- 旧 1.0 collection 不同步新增、删除或重分块变化。

因此回滚到 1.0 后，如果需要检索 1.1 期间新增的资料，必须在旧环境中显式重建旧索引。
