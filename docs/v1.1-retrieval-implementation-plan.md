# 1.1 显式余弦检索实施计划

## 当前实现

项目使用 `SentenceTransformer` 和本地持久化 ChromaDB。`vector_service.py` 负责模型懒加载、编码、资料向量增量写入、删除、后台构建和语义查询。普通检索路由直接调用 `semantic_search`；Agent 的概念工具通过 `citation_service.retrieve_course_chunks` 间接调用同一查询，再对课程、资料和片段执行 SQL 校验并登记请求级引用。

当前资料与查询向量都已归一化，但旧 collection 没有显式距离配置，Chroma 1.5.9 默认使用 squared L2。现有代码用 `1 / (1 + distance)` 生成 `similarity_score`。前端把这个旧转换值显示为百分比。

源码扫描只发现 `vector_service.py` 直接调用 `collection.query`。Agent 中只有 `explain_concept` 进行语义片段检索；其他资料工具是列表、增删等业务查询，不应伪装成向量检索。

## 问题列表

1. collection 的距离空间隐式依赖 Chroma 默认值。
2. `similarity_score` 不是真实余弦相似度。
3. 普通检索和 Agent 引用各自承担一部分校验，结构及异常语义不一致。
4. 普通检索主要信任 Chroma document 和 metadata，未同时用 SQL 片段正文完成二次校验。
5. 没有最低相似度阈值，无关 Top-K 仍可能进入 Agent。
6. collection、模型和环境参数由模块级 `os.getenv` 分散读取，缺少范围校验。
7. 模型/collection 懒加载没有显式并发锁，编码输入和数值结果缺少完整校验。
8. 删除旧向量只按 `material_id`，没有在 Chroma 条件中同时限定 `user_id + course_id`。
9. 重建接口只针对单份资料，没有 1.0 到独立 1.1 collection 的 dry-run、抽样校验和人工切换流程。
10. 检索失败当前可能把内部异常文本返回给前端。

## collection 迁移策略

1. 完整保留旧 `course_material_chunks`，不修改其配置、不删除其记录。
2. 新建 `course_material_chunks_v1_1_cosine`，Chroma 1.5.9 使用官方推荐的 `configuration={"hnsw": {"space": "cosine"}}`。
3. 获取 collection 后读取 `collection.configuration["hnsw"]["space"]`，不是 cosine 就立即拒绝读写。
4. 迁移脚本默认 dry-run，从 SQL 有效片段批量重新编码并以 `chunk_{id}` upsert 到新 collection。
5. 校验数量、配置、向量规范、metadata、抽样查询和作用域过滤后，由用户手工修改配置切换；脚本不写 `.env`。
6. 回滚仅切回旧 collection 名称；不删除任何 collection。

## API 兼容策略

- 保留 `POST /api/search/semantic`、`course_id`、`query`、`top_k` 和旧 `vector_id`。
- 新增响应顶层 `metric="cosine"`、`min_similarity`；结果新增 `course_id`、`course_name`、`file_type`、`similarity_percent`。
- `distance` 明确为余弦距离，`similarity_score` 改为 `clamp(1 - distance, -1, 1)`。
- 不再返回低于阈值的结果；空结果仍为成功响应并显示明确说明。
- Agent citations 保留原字段并使用统一结果中的真实相似度。

## 安全与稳定性

- 所有检索先验证 `Course.id + Course.user_id`。
- 传入资料筛选时验证每个 `Material.id + user_id + course_id`。
- Chroma where 同时包含 `user_id + course_id`，随后 SQL 再校验 `MaterialChunk` 和 `Material` 的用户、课程和关联关系。
- 正文以 SQL `MaterialChunk.content` 为准；不向响应暴露绝对路径、向量、密钥或堆栈。
- 模型和 collection 进程内复用，初始化加锁；请求级引用继续使用现有 `ContextVar` collector。
- Chroma 不可用与“没有相关结果”使用不同异常语义。
- 写入失败只清理当前 1.1 collection 中当前资料的作用域记录并把向量状态标为失败，不触碰旧 1.0 collection。

## 测试策略

- fake 模型验证批量归一化参数、空文本、维度和非有限数检查。
- fake Chroma 验证 cosine 配置、错误 metric 拒绝、where 条件、候选数和 collection 选择。
- SQLite 测试库验证课程、资料、片段的二次权限隔离、metadata 污染过滤、阈值、去重、排序及空结果。
- 数学单元测试覆盖 distance 到 similarity、单位向量余弦关系和浮点截断。
- 验证普通接口和 Agent 引用共用唯一检索函数，低于阈值的片段不会进入 collector。
- fake 批量 writer 验证迁移 dry-run、稳定 ID、重复 upsert、批次部分失败和不删除旧 collection。
- 实际执行 compileall、pytest、脚本 dry-run/离线校验、`npm ci`、`npm run build`、`git diff --check`。

## 回滚策略

代码可切回 `release/1.1.0` 或从 `v1.0.0-retrieval-backup` 建立恢复分支。数据层仅需把 collection 配置改回原名称。旧 collection 始终保留，任何普通升级/回滚脚本都不会调用删除旧 collection。若 1.1 后新增资料需要在旧版本可检索，须在旧环境显式重建这些资料的旧索引。
